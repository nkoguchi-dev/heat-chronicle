# PR レビューガイド（全体共通）

> バックエンドの変更を含む PR は、本ガイドに加えて
> [バックエンド PR レビューガイド](./BACKEND_REVIEW_GUIDE.md) も参照してください。
>
> フロントエンドの変更を含む PR は、本ガイドに加えて
> [フロントエンド PR レビューガイド](./FRONTEND_REVIEW_GUIDE.md) も参照してください。

---

## 1. 全般

- [ ] PR の粒度が適切か（1 つの PR に複数の目的が混在していないか）
- [ ] コミットメッセージが変更内容を簡潔に説明しているか
- [ ] 変更と無関係なファイルが含まれていないか（フォーマッタによる意図しない変更等）
- [ ] 変更の影響範囲が把握されているか（他機能への副作用）
- [ ] 既存の機能を壊していないか（デグレーション）

## 2. 設計・アーキテクチャ

- [ ] 既存のコードベースのパターン・規約と一貫しているか
- [ ] 適切な責務分離がされているか（1 つのクラス・関数が複数の責務を持っていないか）
- [ ] 過剰な抽象化になっていないか（現時点で必要ない汎用化）
- [ ] 逆に、明らかに共通化すべき重複コードがないか
- [ ] 新しいライブラリやパターンの導入に妥当な理由があるか

## 3. コード品質

### 命名

- [ ] 変数・関数・クラスの命名が意図を明確に表しているか
- [ ] 省略しすぎた名前（`tmp`, `val`, `d` 等）を避けているか
- [ ] Boolean 変数・関数に `is_`, `has_`, `can_` 等の接頭辞があるか

### 型安全性

- [ ] 関数の引数と戻り値に型アノテーションがあるか
- [ ] `Any` 型や `any` 型の安易な使用がないか
- [ ] Optional / null / undefined な値の安全なハンドリングがされているか

### 可読性

- [ ] 複雑なロジックに適切なコメントがあるか
- [ ] 1 つの関数が長すぎないか（目安: 50 行以内）
- [ ] ネストが深すぎないか（早期リターンで平坦化できないか）
- [ ] マジックナンバーや意味不明な文字列リテラルが定数化されているか

## 4. テスト

- [ ] 新規・変更された機能にテストが追加されているか
- [ ] 正常系だけでなく異常系（エラーケース）のテストがあるか
- [ ] 境界値のテストがあるか（空文字、0、null、上限値等）
- [ ] テストが他のテストに依存せず独立して実行できるか
- [ ] テストデータが適切に準備・クリーンアップされているか

## 5. セキュリティ

- [ ] 機密情報（API キー、シークレット、パスワード等）がハードコードされていないか
- [ ] 新しい環境変数が適切なファイルに追加されているか
  - 機密情報 → `.env`（Git 対象外）
  - 非機密情報 → `.env.example` 等でサンプルを提供
- [ ] ユーザー入力のバリデーションが適切か
- [ ] XSS やインジェクション攻撃への対策がされているか

## 6. 環境・設定

新しい環境変数の追加がある場合:

- [ ] `.env` の適切な箇所に追加されているか
- [ ] `.env.example`（または `docker-compose.yml`）のサンプルが更新されているか
- [ ] Docker Compose の設定更新が必要ではないか

## 7. パフォーマンス

- [ ] 不要な処理や重複した呼び出しがないか
- [ ] 大量データを扱う処理に適切なページネーション・フィルタリングがあるか
- [ ] 重い処理が適切に非同期化されているか
- [ ] DynamoDB の `scan` を乱用していないか（`query` を優先する）
- [ ] 不要な再レンダリングや API 呼び出しが発生しないか（フロントエンド）

## 8. ドキュメント

- [ ] 変更に応じて CLAUDE.md や README が更新されているか
- [ ] API の変更がある場合、API 仕様（リクエスト・レスポンス）が更新されているか
- [ ] 新しい設定項目や手順が必要な場合、ドキュメントに記載されているか
