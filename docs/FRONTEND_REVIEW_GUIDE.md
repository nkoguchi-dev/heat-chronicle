# フロントエンド PR レビューガイド

> **前提**: 本ガイドはフロントエンド固有の観点をまとめたものです。
> 全体共通の観点（コード品質、セキュリティ、環境変数、テスト品質等）は
> [PR レビューガイド（全体共通）](./REVIEW_GUIDE.md) を参照してください。

このドキュメントは、フロントエンドの PR をレビューする際のチェック観点をまとめたものです。

---

## 1. ディレクトリ構成

- [ ] 機能コードが `src/features/` 以下に配置されているか
- [ ] 汎用 UI コンポーネントが `src/components/ui/` に配置されているか
- [ ] ページコンポーネントが `src/app/` 以下の適切な場所に配置されているか
- [ ] 既存のディレクトリ構成・命名規則と一貫しているか

## 2. 命名規則

- [ ] コンポーネントファイルが PascalCase（例: `HeatMap.tsx`）になっているか
- [ ] ユーティリティ・ライブラリファイルが kebab-case（例: `api-client.ts`）になっているか
- [ ] カスタムフックが `use-xxx.ts` 形式になっているか（例: `use-temperature.ts`）
- [ ] 型定義ファイルが適切な名前になっているか

## 3. API クライアント

- [ ] バックエンド API へのリクエストが `src/features/shared/libs/api-client.ts` 経由で行われているか（直接 `fetch` しない）
- [ ] エラーハンドリングが適切に行われているか
- [ ] レスポンスの型が `src/types/` の型定義と整合しているか

## 4. 型安全性

- [ ] `src/types/` の型定義がバックエンド API のレスポンス仕様と整合しているか
- [ ] `any` の安易な使用がないか（型推論・型定義で代替できるか）
- [ ] `null` / `undefined` な値の安全なハンドリングがされているか
- [ ] コンポーネントの Props に適切な型定義があるか

## 5. スタイリング

- [ ] Tailwind CSS v4 のユーティリティクラスを使用しているか
- [ ] インラインスタイル（`style={{ }}`）を避け、Tailwind クラスで代替できているか
- [ ] 新しい UI コンポーネントに shadcn/ui が活用されているか
- [ ] レスポンシブデザインが考慮されているか（必要な箇所に `sm:`, `md:`, `lg:` 等のブレークポイントがあるか）

## 6. パフォーマンス

- [ ] 不要な再レンダリングが発生していないか（`useMemo`, `useCallback` の適切な使用）
- [ ] `useEffect` の依存配列が正しく設定されているか
- [ ] Canvas 描画（ヒートマップ）の更新が最小限か（データ変化時のみ再描画する）
- [ ] 大量データを一度に描画していないか（必要に応じて仮想化・分割ロードを検討する）
- [ ] 静的エクスポート（`next export`）に非対応の Next.js 機能を使用していないか

## 7. Next.js App Router

- [ ] Server Component と Client Component の使い分けが適切か
- [ ] `"use client"` ディレクティブが必要な箇所にのみ付与されているか
- [ ] データフェッチがサーバーサイドで完結できる場合、Client Component を避けているか
- [ ] 静的エクスポート前提のため、動的ルーティング（`generateStaticParams`）が正しく設定されているか

## 8. ビルド確認

以下のコマンドが全て通ることを確認する:

```bash
npm run lint
npm run build
```

- [ ] `npm run lint`（ESLint）が通るか
- [ ] `npm run build`（Next.js ビルド）がエラーなく完了するか
- [ ] TypeScript のコンパイルエラーがないか
